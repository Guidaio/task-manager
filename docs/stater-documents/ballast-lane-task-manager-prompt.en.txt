DESIGN DECISIONS AND RATIONALE
Please create docs/design-decisions.md and keep it updated.

Use the following rationale as the initial decision record.

1. Why Task Manager?
We selected a Task Manager because the exercise itself mentions a task management system as the GenAI example, including CRUD operations, title, description, status, due date, and user association. This keeps the use case aligned with the assignment while still allowing us to demonstrate authentication, authorization, validation, data access, testing, frontend integration, and clean architecture.

2. Why a single repository?
The submission instructions ask for only one GitHub link and one repository. A monorepo with backend, frontend, and docs keeps the submission easier to review, run, and present.

3. Why .NET 8 + ASP.NET Core Web API with Controllers?
The role focuses on modern .NET and the test asks for ASP.NET Web API. Controllers make the API layer explicit and easy to review, especially for authentication, authorization, validation, HTTP verbs, request/response contracts, and code review discussion.

4. Why Clean Architecture?
The test explicitly evaluates Clean Architecture, separation of concerns, and independence of components. The solution should separate Domain, Application, Infrastructure, and API responsibilities so business logic does not depend directly on the database, framework, or HTTP layer.

5. Why ADO.NET?
The test forbids Entity Framework, Dapper, and Mediator. ADO.NET with parameterized SQL allows us to implement the required custom data access layer while respecting the constraint. It also makes SQL behavior explicit and demonstrates understanding of database access without an ORM.

6. Why SQL Server?
SQL Server is a strong fit for a .NET backend project and aligns well with enterprise .NET experience. It also works well with Docker Compose and ADO.NET.

7. Why Angular?
The position is full stack and accepts modern frontend frameworks. Angular is a good fit for a .NET enterprise-style application and allows us to demonstrate services, components, HttpClient, routing, forms, JWT interceptor, guards, state handling, and frontend/backend integration.

8. Why FluentValidation?
FluentValidation gives clear request validation without mixing validation rules into controllers. It keeps input validation organized and testable. Business rules should still remain in the Application/Domain layer.

9. Why JWT authentication?
The test requires user creation, login, authorized endpoints, and non-authorized endpoints. JWT is a standard and practical approach for protecting API endpoints in a stateless web API.

10. Why SignalR + BackgroundService + Channel<T>?
The user experience benefits from real-time feedback after task operations. SignalR allows the backend to notify the Angular frontend without requiring the user to manually check updates. The in-memory Channel<T> and BackgroundService decouple the main task operation from notification delivery while keeping the project runnable locally and avoiding unnecessary external cloud dependencies.

11. Why not Azure Service Bus?
A durable broker such as Azure Service Bus would be a valid production evolution, but it adds cloud setup, secrets, and external dependencies. For this test, SignalR plus an in-memory background queue demonstrates asynchronous processing and full stack interaction without making local execution harder for reviewers.

12. Why Docker Compose?
Docker Compose makes it easier for reviewers to run the SQL Server dependency locally and improves the professionalism of the setup. Backend/frontend containers can be added if feasible, but the minimum useful Docker setup is SQL Server.

13. Why unit and integration tests?
The assignment explicitly asks for unit tests and says TDD is preferable. Unit tests validate business rules and services in isolation. Integration tests validate the API pipeline, authentication, authorization, database behavior, and main user flows.

14. Why document AI usage?
The assignment explicitly evaluates GenAI usage, prompt engineering, and critical thinking when reviewing AI-generated code. Therefore, all meaningful prompts, outputs, corrections, validations, and human decisions should be documented.

SIGNALR + BACKGROUND QUEUE
Goal:
When a user creates, updates, or deletes a task, the API should complete the main operation and enqueue a notification.
A BackgroundService processes that queue and sends the message to the frontend through SignalR.

Flow:
1. User performs an action in Angular.
2. API validates the request.
3. Application service executes the rule.
4. Repository persists the data.
5. Notification is persisted or created.
6. A notification message is enqueued in Channel<T>.
7. Endpoint returns the response.
8. BackgroundService consumes the queue.
9. SignalR sends the notification to the connected user.
10. Frontend shows a toast/banner.

Important:
- SignalR should be used for real-time notification.
- The queue can be in-memory with Channel<T>.
- Do not use real Azure Service Bus in this test.
- Document that in production, the in-memory queue could be replaced with Azure Service Bus, RabbitMQ, or another durable broker.
